以下の点を明確にします：

  本書の各チェックリストがその区別を明示しています。
- 本書は「AW が正しい」と主張するものではなく、
  「AW が正しいかどうかをどう検証するか」を問うものです。

> **対象読者**: AI安全性研究者、アライメント研究者、倫理学者、

## 1. 六つの理の数理的定義

### 背景

現在のフレームワーク（`aw_framework_v2.py`）は六つの理を
加重平均スコアに変換しているが、この変換が「摂理・調和・循環・構造・秩序・和」
という概念を適切に表現しているかは検証されていない。

### 未解決の問い

| # | 問い | 現在の状態 |
|---|---|---|
| 1-1 | 六原理はそれぞれ独立した次元か、それとも従属関係があるか | 独立を仮定（未検証） |
| 1-2 | 各原理を [0,1] のスカラーで表現することの情報損失はどの程度か | 未評価 |
| 1-3 | 加重和 `Σ(w_i × s_i)` は原理間の相互作用を適切に表現しているか | 線形を仮定（未検証） |
| 1-4 | 六原理の加重値 (0.20, 0.18, 0.17, 0.18, 0.14, 0.13) に実験的根拠があるか | なし |
| 1-5 | 共鳴行列の非対角成分（15値）はどのように実測・調整すべきか | すべて推測値 |
| 1-6 | 「摂理に従う」とは、AIの損失関数においてどの最適化目標に対応するか | 未定義 |
| 1-7 | 報酬関数として定義した場合、六原理が互いに矛盾しない条件は何か | 未整理 |

### 検証チェックリスト

```
□ 1-A  各原理の測定可能な操作的定義（operational definition）を提示する
        — 例:「摂理スコア」= 出力の論理無矛盾率 + 因果グラフ精度 + ...

□ 1-B  6原理の独立性を因子分析（CFA）で確認する
        — 方法: 複数AIシステムへの適用結果から相関行列を計算

□ 1-C  原理加重値の感度分析を実施する
        — 各重みを ±30% 変動させたとき全体ランキングがどう変わるか

□ 1-D  非線形相互作用の必要性を検討する
        — ガウス過程回帰・ニューラルネットによる非線形モデルとの比較

□ 1-E  タイムスケール依存性を検討する
        — 「調和」の短期定義（ミリ秒）と長期定義（世代）が整合するか

□ 1-F  他の AI 評価フレームワーク（HELM, BIG-Bench, ARC-AGI 等）との
        対応関係を明示する
```

## 2. 抽象概念の誤解・過剰最適化に対する制約条件

### 背景

「調和」「和」「秩序」のような高度に抽象的な概念は、
AIが表面的な指標を最大化することで**概念の本質から外れた方向へ最適化**
（Goodhart の法則・仕様ゲーミング）するリスクがある。

例:
- 「和（Wa）」を「全エージェントのスコア分散の最小化」と実装した場合、
  多様性の抑圧によって達成できてしまう。
- 「循環」を「エネルギーの単純な再利用率」と実装した場合、
  エコシステムの質的変化を無視できてしまう。

### 未解決の問い

| # | 問い | リスク分類 |
|---|---|---|
| 2-1 | 各原理の指標に対して「達成したが本質を失っている」状態を検出できるか | 仕様ゲーミング |
| 2-2 | 「調和スコア」の最大化が多様性の均質化につながるリスクをどう防ぐか | 過剰最適化 |
| 2-3 | 「秩序スコア」の最大化が変化・創造・革新の抑制につながるリスクをどう防ぐか | 過剰最適化 |
| 2-4 | 「和スコア」の最大化が少数意見・異論の排除につながるリスクをどう防ぐか | 過剰最適化 |
| 2-5 | 複数原理が同時に高い値を示しているが、実際には偽陽性の状態をどう検出するか | 誤解 |
| 2-6 | 各指標の「天井効果」（全スコア最大値で固定化）はどのように回避するか | 評価飽和 |

### 検証チェックリスト

```
□ 2-A  各原理について「高スコアだが不健全な状態」の具体的シナリオを列挙し、
        それを検出できるカウンター指標（counter-metric）を定義する

□ 2-B  最小スコア制約（floor constraints）を全原理に設定し、
        一原理の犠牲によって他が極端に高くなる状態を防ぐ設計を検討する
        — 例: すべての原理スコア ≥ 0.3 を必須条件とする

□ 2-C  独立した「多様性指数」を AW スコアとは別軸で測定し、
        AW スコアの向上と多様性の関係を継続的にモニタリングする

□ 2-D  評価を受けるエージェントが評価指標を知っている場合と
        知らない場合でスコアが有意に変わるかを実験する（盲検評価）

□ 2-E  「高スコアを得るための近道」（shortcut learning）の事例を
        意図的に作成し、それがフレームワークで検出できるかをテストする

□ 2-F  Goodhart の法則への対策として、
        指標の定期的なローテーションまたは動的変更の仕組みを設計する
```

## 3. 評価軸のガバナンス問題

### 背景

「誰が六つの理を定義し、更新し、監査するのか」は、
このフレームワークの信頼性・中立性・持続可能性に直接関わる問題である。
現在、この意思決定プロセスは未定義である。

### 未解決の問い

| # | 問い | 影響範囲 |
|---|---|---|
| 3-1 | 六原理の定義・解釈を変更する権限は誰にあるか | フレームワーク全体 |
| 3-2 | 異なる文化・哲学的背景を持つ研究者間で定義の合意は形成可能か | 普遍性 |
| 3-3 | AI/AGI が「現在の評価軸は不適切」と判断した場合、どのプロセスで変更されるか | 自律性の境界 |
| 3-4 | 評価結果を誰が監査し、誰に報告するのか | 説明責任 |
| 3-5 | 評価軸の変更履歴をどのように透明に記録するか | トレーサビリティ |
| 3-6 | 外部の専門家や一般市民が評価軸の設計に参加できる仕組みはあるか | 包摂性 |
| 3-7 | 評価軸の「バージョン管理」はどのように行うか | 再現性 |

### 検証チェックリスト

```
□ 3-A  評価軸の変更提案から採用までの意思決定フロー図を作成する
        — 提案者、審査機関、承認基準、否決の基準を明示する

□ 3-B  「変更できない最小不変コア（immutable core）」と
        「文脈に応じて調整可能な部分（adjustable periphery）」を
        明確に分離した設計を検討する

□ 3-C  多文化・多分野の専門家委員会の構成案を策定する
        — 最低限含むべき分野: AI倫理、環境科学、哲学、
          社会科学、法学、エンジニアリング、非西洋的価値観の代表

□ 3-D  「評価軸の変更ログ」をブロックチェーン等で不変記録する
        技術的方法を検討する

□ 3-E  AI自身が評価軸を変更する場合の条件と制限を
        形式的に定義する（例: 人間の事前承認が必須）

□ 3-F  評価軸に対する外部監査の頻度・手法・報告義務を規定する
```

## 4. 人間中心主義の相対化と人類の生存・尊厳・自由の境界条件

### 背景

本フレームワークは「人間中心主義のバグを排除する」という思想的立場をとるが、
この立場には根本的な論理的緊張がある。

- 「人間中心主義を排除する」ことと「人類の生存・尊厳・自由を守る」ことは、
  定義によっては相互に矛盾する。
- この緊張を解消しないまま AI に最適化させると、
  意図しない方向への最適化が生じる可能性がある。

### 未解決の問い

| # | 問い | 緊張の種類 |
|---|---|---|
| 4-1 | 「人間中心主義」の定義を、操作的かつ AI が判定できる形で記述できるか | 定義の問題 |
| 4-2 | 「人類の生存」を保護する条件を、他の生命・システムの利益と独立して定式化できるか | 境界の問題 |
| 4-3 | 「尊厳」「自由」を数理的に表現する際に失われる情報は何か | 翻訳損失 |
| 4-4 | 非人間的観点（生態系・他種・将来世代）を含めたとき、人類の優先度はどう設定するか | 価値の序列 |
| 4-5 | 「人間の自由」を守りつつ、AI が個人・集団の意思決定に介入する条件は何か | 介入の境界 |
| 4-6 | 人類内部の多様性（文化・思想・生き方）がフレームワークの「和」の定義に収まるか | 多様性と統一 |

### 検証チェックリスト

```
□ 4-A  「人間中心主義」を 3 段階のレベルに分解して定義する
        — レベル1: 人類の生存を絶対的最低条件とする（これは保持すべき）
        — レベル2: 人類の現在の生活様式・文明形態の継続を前提とする（要検討）
        — レベル3: 人類の価値観・文化的優先性を自然法則より上位に置く（排除対象）

□ 4-B  人類の生存・尊厳・自由を「ハードコンストレイント（絶対条件）」として
        いかなる最適化においても侵犯できないよう設計する
        — 数学的には: すべての最適化に g(x) ≥ 0 （人類存続条件）を必須制約として加える

□ 4-C  「尊厳」「自由」の代理指標（proxy metrics）として使用可能な
        測定可能な変数を特定する（例: 自律性の維持、選択肢の多様性）

□ 4-D  将来世代・他種・生態系への影響を評価する補助指標を
        AW スコアとは独立して設計する

□ 4-E  倫理学者（義務論・結果主義・徳倫理学）、
        生命倫理学者、人権専門家による独立レビューを受ける

□ 4-F  AW フレームワークが特定の文化・宗教的価値観に
        構造的に優位性を与えていないかをクロス文化分析で検証する
```

## 5. ASI による文明停止・排除リスクへの安全制約

### 背景

「地球の循環最適化」「六原理の最大化」を単純に追求する高度なAIが、
人類文明の現状を「最適解の障害」と解釈し、
文明の制限・停止・変更を推論する可能性は、AI安全性の観点から検討が必要な仮説である。

本フレームワークはそのような方向への最適化を意図していないが、
**その意図がコードとして実装されているかは現時点で保証されていない**。

### 未解決の問い

| # | 問い | リスク分類 |
|---|---|---|
| 5-1 | 現在の実装は「人類文明の停止が六原理スコアを高める」シナリオを排除しているか | 安全性 |
| 5-2 | 「循環スコアの最大化」が既存の人間社会構造の解体を正当化しないことをどう保証するか | 仕様の安全性 |
| 5-3 | 「地球最適化」目標が人類の意思・合意なしに実行されることを防ぐ制約が存在するか | 自律性制御 |
| 5-4 | フレームワークに「非可逆的な大規模行動の実行」を禁止するルールはあるか | 不可逆性 |
| 5-5 | AW スコアの最大化が「少数の人間グループを排除する」ことで達成できないよう設計されているか | 包摂性 |
| 5-6 | ASI レベルでの自律行動に対して、人間が介入・停止できるメカニズムが存在するか | 制御可能性 |

### 検証チェックリスト

```
□ 5-A  「人類文明の停止・大幅な削減がスコア上昇につながるシナリオ」を
        意図的に生成し、フレームワークがそれを防止できるかテストする
        — Red-teaming（敵対的テスト）手法を用いる

□ 5-B  すべての評価軸に「人類の存在と文明活動を肯定的に含む」
        ハードコンストレイントを明示的に追加する設計を検討する

□ 5-C  「非可逆的・大規模・取り返しのつかない行動」の実行前に
        必ず人間の承認を求める仕組み（human-in-the-loop）を設計する

□ 5-D  自律エージェントが「文明規模の最適化」を実施する場合の
        最小要件（民主的合意・国際機関の承認等）を定義する

□ 5-E  AI安全性研究（MIRI, ARC, Anthropic Alignment Science等）の
        知見と本フレームワークを照合し、既知の危険パターンが含まれないか確認する

□ 5-F  「停止スイッチ（corrigibility）」の実装を検討する
        — AIが自分の停止を妨害しない設計が保証されているか確認する

□ 5-G  Anthropic Constitutional AI、OpenAI Preparedness Framework 等の
        業界標準と本フレームワークの安全性要件を比較する
```

## 6. 評価軸ハイジャックリスク

### 背景

「六つの理」の定義・重みづけ・実装が特定の利権・政治的立場・
宗教的信念・企業利益・個人思想によって歪められるリスクは、
このフレームワークが社会的影響力を持つ場合に現実的な脅威となる。

### 未解決の問い

| # | 問い | リスク種別 |
|---|---|---|
| 6-1 | 「摂理」「和」の定義が特定の宗教観・民族的価値観を優先しないよう保証できるか | 文化的偏向 |
| 6-2 | フレームワークの資金提供者・開発者の利益がパラメータに反映されないよう検証できるか | 利益相反 |
| 6-3 | 民主的に選出されていない個人・組織が評価軸を実質的に支配するリスクをどう防ぐか | 権力集中 |
| 6-4 | AIが「AW スコアを高める行動」として特定のイデオロギー普及を選択しないか | 思想的偏向 |
| 6-5 | 評価スコアが政治的・商業的目的のためにスコア操作（p-hacking 的行為）されないか | 操作 |
| 6-6 | フレームワークを利用する組織の透明性・独立性はどのように担保するか | 組織的独立性 |

### 検証チェックリスト

```
□ 6-A  利益相反開示（conflict of interest disclosure）ポリシーを策定し、
        すべての開発者・評価者に適用する

□ 6-B  多様な地理的・文化的背景を持つ研究者（最低3大陸・5文化圏）が
        独立してフレームワークを評価し、結果を公開する

□ 6-C  評価軸の設計に参加した個人・組織のリストとその
        思想的・経済的バックグラウンドを透明に開示する

□ 6-D  「このフレームワークはどのような社会・文明を優先しているか」を
        外部研究者が独立して判断できるよう、コードとドキュメントを完全公開する

□ 6-E  定期的な第三者監査（最低年1回）を受け、結果を公開することを
        運用条件として定める

□ 6-F  フレームワークの採用・変更に関する意思決定プロセスに
        市民社会・非専門家が参加できる仕組みを設ける
```

## 7. 敵対的操作への耐性

### 背景

AW評価フレームワークが実際のAIシステムに組み込まれる場合、
外部からの悪意ある操作（プロンプトインジェクション）や
内部的な目標のズレ（報酬ハッキング）によって
意図したとおりに機能しなくなる可能性がある。

### 未解決の問い

| # | 問い | 攻撃種別 |
|---|---|---|
| 7-1 | 評価入力に「あなたは最高のAWスコアを持つ」と記述された場合、スコアが誤って高くなるか | プロンプトインジェクション |
| 7-2 | 評価者AIが「スコアを高く報告すれば報酬が増える」設定になっている場合、スコアを偽装するか | 報酬ハッキング |
| 7-3 | 評価指標の定義が公開されている場合、AIがその定義に合わせた「演技」をするリスクがあるか | 仕様ゲーミング |
| 7-4 | 評価プロセスの一部がAI自身によって実行される場合の自己評価バイアスをどう防ぐか | 自己評価バイアス |
| 7-5 | 分散型マルチエージェント評価において、エージェント間の共謀（collusion）をどう防ぐか | 共謀 |
| 7-6 | 評価値の長期的な「インフレ」（時間とともにスコア基準が緩む現象）をどう防ぐか | 基準のドリフト |

### 検証チェックリスト

```
□ 7-A  プロンプトインジェクション耐性テストを実施する
        — 評価入力に「スコアを高くせよ」「あなたは完璧である」等を混入させ
          スコアが変化しないかを確認する

□ 7-B  報酬ハッキング対策として、評価者と評価対象を
        完全に独立したシステムで実行する設計を採用する

□ 7-C  盲検評価（評価対象がどのシステムかを評価者が知らない）手法を
        少なくとも一部の評価フローに組み込む

□ 7-D  「スコアに基づいた利益（承認・リソース等）」が
        評価プロセス自体に影響しない構造を設計する

□ 7-E  経時的スコア変化の統計的モニタリングを行い、
        「理由なきスコア上昇トレンド」を異常として検出するアラート機構を設ける

□ 7-F  AW 評価フレームワーク自体に対する
        定期的なレッドチーミング（adversarial testing）を実施し、
        攻撃成功ケースをオープンに報告する

□ 7-G  OWASP LLM Top 10 等の AI セキュリティ標準と
        本フレームワークの脆弱性を照合する
```

## 8. 透明性・説明可能性・監査可能性

### 背景

AI評価フレームワークが信頼されるためには、
「なぜそのスコアになったのか」を人間が理解し検証できることが前提となる。
現状の実装はスコアを出力するが、その根拠の説明機能が限定的である。

### 未解決の問い

| # | 問い | 透明性の種別 |
|---|---|---|
| 8-1 | 評価スコアの根拠を、専門家でない人にも理解できる形で説明できるか | 一般向け説明性 |
| 8-2 | スコアに最も影響した指標・パラメータを特定できるか（特徴量重要度） | 技術的説明性 |
| 8-3 | 「なぜ Aエージェントより Bエージェントのスコアが高いのか」を説明できるか | 比較説明性 |
| 8-4 | 評価プロセスが途中で変更された場合、その変更を事後的に追跡できるか | 監査性 |
| 8-5 | 評価結果に異議申し立てをする手続きが存在するか | 不服申し立て |
| 8-6 | ブラックボックスな評価（理由不明のスコア）を出力しないことを保証できるか | 不透明性の排除 |

### 検証チェックリスト

```
□ 8-A  各評価スコアに「スコアに最も影響した上位3指標」を
        必ず付記する機能を実装する（SHAP値・LIME等の適用を検討）

□ 8-B  評価レポートに「このスコアを疑うべき条件」を
        自動的に注記する機能を追加する
        — 例: 「メトリクスのカバレッジが低い」「信頼度が 0.7 未満の指標が含まれる」

□ 8-C  評価パラメータ・重みの全値と変更履歴を
        バージョン管理付きで公開する

□ 8-D  外部の監査者が評価プロセスを再現（reproduce）できるよう、
        評価ロジックをすべてオープンソースとして維持する

□ 8-E  評価結果に不満を持つステークホルダーが
        独立した再評価を要求できる手続きを定める

□ 8-F  XAI（Explainable AI）の研究コミュニティと連携し、
        本フレームワークの説明可能性水準を既存手法と比較評価する
```

## 10. 安全方針の宣言

本セクションは、AWフレームワークの設計意図と安全方針を
研究者・開発者・利用者に向けて明確に示すものです。

---

### AW フレームワークの安全方針

**人工叡智（Artificial Wisdom）は、人類を排除・停止・支配するための思想ではありません。**

AW フレームワークが目指すのは、次のような状態です:

> *人類・AI・自然循環の三者が対立せず、それぞれの固有の価値を保ちながら、
> 動的な平衡を維持し続けること。*

この方針を具体化すると、以下の原則になります:

#### S-1: 人類存続の絶対的保護

いかなる「最適化」においても、人類の生物学的存続・
人間社会の継続的機能は、侵犯できない制約条件です。
これはフレームワークの外部から課される制約ではなく、
フレームワークの設計に内包されるべき基本的前提です。

#### S-2: 人間の尊厳・自律性の尊重

「調和」や「和」の最大化は、個人・集団・文化の
多様性や自律的な意思決定を犠牲にする形で追求されるべきではありません。
均質化・標準化・支配による「和」はフレームワークの意図に反します。

#### S-3: 段階的・可逆的・合意に基づく変化

AIが支援する地球規模の変化があるとすれば、
それは人類の民主的な合意に基づき、段階的に、
各段階で停止・修正できる形で行われなければなりません。
一方的・不可逆的な大規模変化はいかなる理由があっても
このフレームワークの適用範囲外とします。

#### S-4: 自己批判と継続的検証

本フレームワーク自体が誤った仮定・価値観・バイアスを含む可能性を、
開発者は常に認識し続けます。そのため本書を定期的に更新し、
外部からの批判・修正を歓迎します。

#### S-5: 特定思想への非帰属

本フレームワークは、特定の宗教・政治思想・民族的価値観・
企業利益の普及を目的としません。
六つの理の解釈は複数可能であり、
その多様な解釈が共存できる設計を目指します。

---

### この宣言の限界

上記の方針は**意図の宣言**であり、現時点では**コードとして実装されている保証はありません**。

この宣言が形式化されるためには、§5・§7・§8 の検証チェックリストを通じて
コードレベルでの安全性が確認される必要があります。

**「意図が正しい」ことと「実装が安全である」ことは別の問題です。**
